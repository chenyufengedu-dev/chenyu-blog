# Cyber Office — 嵌入个人博客的多 Agent 协作实验室

> 设计文档 · 2026-06-14
> 路由：`/cyber-office`

## 1. 背景与目标

在 Chenyu 的个人技术博客（Next.js + Tailwind，部署于 Vercel）中嵌入一个可交互的**多 Agent 协作实验室**。用户提出一个议题，多个角色 Agent 围坐圆桌、由主持人 Agent 动态调度轮流发言、互相补充与争论，最后由总结 Agent 产出结构化结论（如文章大纲）。

**目的**：
- 锻炼并展示 **Agent 编排能力**（动态调度、受约束指令、多智能体协作），而非简单的单次 LLM 调用。
- 作为求职实习的作品集亮点：与本人科研背景（生信/空间转录组）强相关，且可在网页中直接体验。

## 2. 核心决策（已确认）

| 维度 | 决策 |
|---|---|
| 嵌入方式 | 现有 Next.js 博客新增路由 `/cyber-office` |
| LLM 提供方 | 后端使用本人 Anthropic API Key（仅存于服务端） |
| 安全模型 | **回放优先 + 实时需限流**：默认播放预生成真实会议；「实时运行」受限流/单次封顶/每日预算保护 |
| 角色范围 | **预设为主 + 轻量自定义**（预设 5-6 个角色，允许改名 + 一句话人设微调） |
| 编排模式 | **主持人 Agent 动态调度**：主持人每轮输出结构化指令决定下一个发言者或进入总结 |
| 视觉风格 | **俯视圆桌 + 星露谷风格像素小人**，主持人逐个点名，角色举手起身发言 |
| 像素美术起步 | **占位方块/圆形**先跑通逻辑（选项 c），美术最后替换 |

## 3. 整体架构

```
浏览器 (React 场景 + 像素动画)
        │  SSE / 流式
        ▼
Next.js Route Handler (/api/cyber-office/run)   ← Anthropic Key 仅在此
        │
        ▼
编排器 Orchestrator (服务端，纯 TS 逻辑)
        │  循环调用
        ▼
Anthropic API (主持人 / 各角色 / 总结 Agent)
```

**关键原则**：API Key 与编排逻辑全部在服务端；前端只负责"演"——消费服务端推送的事件流驱动像素动画。前后端通过一套统一的**事件流**解耦。

## 4. 编排核心

一次圆桌会议是一个**主持人驱动的循环**：

1. 用户提交议题 + 勾选参会角色。
2. 主持人 Agent 接到议题 → 输出开场白 + 第一个点名指令。
3. 被点名角色 Agent 发言（上下文 = 议题 + 已有发言记录）。
4. 主持人 Agent 读取最新发言 → 决定：继续点名 / 让某角色反驳 / 进入总结。
5. 重复 3-4，直到主持人判定讨论充分或达到**轮数硬上限（默认 8-10 次发言）**。
6. 总结 Agent 汇总全场 → 产出结构化结论（如文章大纲）。

**主持人输出受约束的结构化指令（JSON）**，而非自由文本，例如：

```json
{ "action": "call_on", "speaker": "frontend", "prompt": "从可视化实现角度说说…" }
{ "action": "summarize" }
```

编排器解析该指令决定下一步。这是本项目最核心的能力展示点（director agent 做动态调度 + 受约束的工具调用式输出）。

**角色 system prompt**：每个角色 = 人设 + 职责 + 约束（"只从自身专业角度发言、简洁"）。

## 5. 事件流 / 数据模型

服务端通过 **SSE（Server-Sent Events）** 流式推送事件，前端按事件演动画：

```ts
type OfficeEvent =
  | { type: "meeting_start"; topic: string; participants: Role[] }
  | { type: "host_speak"; text: string }             // 主持人开场/串场
  | { type: "call_on"; speaker: RoleId }              // 点名 → 小人举手起身
  | { type: "speaking_start"; speaker: RoleId }       // 开始说 → 气泡出现
  | { type: "token"; speaker: RoleId; delta: string } // 流式文字逐字进气泡
  | { type: "speaking_end"; speaker: RoleId }         // 说完 → 坐下
  | { type: "summary"; outline: string }              // 总结 Agent 产出
  | { type: "meeting_end" }
  | { type: "error"; message: string };
```

前端维护 `roles[]` 状态（每角色：座位位置、当前状态 `idle | raising_hand | speaking | thinking`、当前气泡文本），事件到来时更新状态并触发动画。

**回放模式**即一段预先录制的 `OfficeEvent[]` JSON，前端用同一套消费逻辑播放——回放与实时共用前端代码。

## 6. 前端场景

- **场景**：俯视圆桌，N 个座位用极坐标均匀分布在桌周；中央放议题卡片。
- **像素小人**：每角色一张 sprite，至少 3 帧动作（坐着 / 举手 / 说话）。用 CSS `steps()` 或帧切换实现。起步用纯色方块占位。
- **动作映射**：`call_on` → 举手帧 + 轻微上移；`speaking` → 说话帧 + 头顶气泡（流式逐字）；`speaking_end` → 回坐。
- **移动端**：圆桌整体缩放；过挤时气泡降级为底部统一对话条，但小人状态点保留，保证手机可用。
- **设计一致性**：像素风是该组件的"破例特区"，但外壳 UI（按钮、面板、总结区）仍遵循博客的极简橙色系（`#ea580c`）。

## 7. 安全措施

- **默认回放**：内置 1-2 段预生成真实会议录像，进页面自动播放，零成本。
- **「▶ 实时运行」三重保护**：
  1. **限流**：基于 IP + 会话计数（Vercel KV / Upstash Redis），如每人每小时 N 次、全站每分钟 M 次。
  2. **单次封顶**：发言轮数上限 + 每个 Agent `max_tokens` 限制。
  3. **每日总预算**：累计 token 超阈值当天自动降级回放，提示"今日体验额度已用完"。
- **输入校验**：议题长度限制；可选敏感内容过滤。

## 8. 模型与成本

- **角色 Agent**：Claude Haiku 4.5（快、便宜，适合多次调用）。
- **主持人 / 总结 Agent**：Claude Sonnet 4.6（调度与收口需更强推理）。
- 单次会议（约 8 轮）成本约几分钱人民币级别；配合限流与每日封顶，月成本可控在很低水平。

## 9. 分阶段实现路线

每阶段均为可独立运行、可见成果的里程碑：

| 阶段 | 目标 | 产出 |
|---|---|---|
| **P0** | 静态场景 | 俯视圆桌 + 占位小人（方块）布局，纯前端无交互 |
| **P1** | 事件回放引擎 | 写死一段 `OfficeEvent[]`，前端能按事件演：点名→举手→气泡逐字→坐下。**不接 API** |
| **P2** | 接真实编排（无安全） | 后端 Orchestrator + 主持人/角色/总结 Agent，SSE 流式推事件，本地 `.env` Key，能真实跑一场会 |
| **P3** | 安全 + 回放默认 | 加限流、单次封顶、每日预算；默认回放、按钮触发实时 |
| **P4** | 像素美术 + 动效打磨 | 替换占位为星露谷风 sprite，举手/说话动画，气泡、音效（可选） |
| **P5** | 自定义 + 收尾 | 轻量自定义角色名/人设、总结结果导出（复制 Markdown）、移动端降级、嵌入博客导航 |

**P1 是关键解耦点**：先把"演"（前端动画）与"算"（后端 Agent）通过事件流分开，二者可独立开发、互不阻塞。

## 10. 暂不纳入范围（YAGNI）

- 多人实时协作 / 多用户房间。
- 会议历史持久化与账户系统（除回放样本外不存储用户会议）。
- 语音 / TTS。
- 角色头像完全自由上传（仅预设 + 轻量文字自定义）。

## 11. 待后续阶段细化的开放项

- sprite 美术的最终来源（CC0 素材 vs 自绘），P4 时决定。
- 限流具体阈值与每日预算数值，P3 时根据实际定价标定。
- 总结产物的结构化格式（纯 Markdown 大纲 vs 更结构化字段），P2 时随主持人/总结 prompt 一并定。
